Introduction

The Boundary Service API provides a restful interface to a wealth of data about official geographic boundaries within Illinois. Data is retrieved from the API as JSON (optionally JSONP) with nested GeoJSON for describing the boundary shapes.

There are two resources described by the API: Boundary Sets and Boundaries.

Boundary Sets

To see a paginated list of available Boundary Sets access the following url:

http://{{ API_DOMAIN }}/1.0/boundary-set/

Boundary Sets describe the extent and provenance of the data as well how recently it has been updated. A complete list of attributes for a Boundary Set can be introspected by accessing:

http://{{ API_DOMAIN }}/1.0/boundary-set/schema/

A single BoundarySet may be accessed directly by its slug:

http://{{ API_DOMAIN }}/1.0/boundary-set/community-areas/

Boundaries

Resources representing individual Boundaries can be accessed at:

http://{{ API_DOMAIN }}/1.0/boundary/

As with Boundary Sets a description of all attributes returned with the Boundary can be retrieved from:

http://{{ API_DOMAIN }}/1.0/boundary/schema/

A single Boundary may be accessed directly by its slug:

http://{{ API_DOMAIN }}/1.0/boundary/austin-community-area/

For purposes of easy access the Boundary Service exposes a second non-canonical path to any Boundary by appending the Boundary's "external_id" property to the BoundarySet's uri. Example:

http://{{ API_DOMAIN }}/1.0/boundary-set/wards/22

This returns the 22nd Chicago city ward.

Limiting resources returned

Pagination of returned resources is handled with the page and limit parameters.

In addition, Boundaries may (and should) be constrained by passing a comma-delimited list of BoundarySet slugs to the sets parameter:

http://{{ API_DOMAIN }}/1.0/boundary/?sets=community-areas,neighborhoods

Point queries

The fundamental query exposed by the Boundary Service is a point-in-polygon search by latitude and longitude. Here is an example:

http://{{ API_DOMAIN }}/1.0/boundary/?contains=41.8903239,-87.623458&sets=community-areas

This will return the community area that Tribune Tower resides in, the Near North Side.

Notice that in this case we also pass the set parameter to constrain the query. For practical purposes its important to limit your queries to only the data you need as it is much more performant than querying and returning all the data.

Near queries

The near query will find all Boundaries that intersect a circle of a specified radius using a given point as its center:

http://{{ API_DOMAIN }}/1.0/boundary/?near==41.8903239,-87.623458,1mi&sets=wards

This returns all wards within one mile of Tribune Tower. The radius parameter may be specified in any units supported by GeoDjango's Measurement objects

Please note this query will execute much more slowly than the point query.

Intersection queries

The intersects query find all boundaries in the Boundary Service which intersect another. To find all neighborhoods which intersect the Humbolt Park Community Area, one would retrieve:

http://{{ API_DOMAIN }}/1.0/boundary/?intersects=humboldt-park-community-area&sets=neighborhoods

The Boundary Service does not currently support querying by arbitrary shapes.

JSONP

For use in client-side applications the Boundary Service API may be accessed via JSONP. Simply append the format and callback parameters to the query string:

http://{{ DOMAIN_API }}/1.0/boundary/?format=jsonp&callback=func

Throttling & API keys

The Boundary Service API can be accessed anonymously up to 100 times per hour per IP address. This throttling is in place in order to ensure that it can be scaled to meet demand without extensive outages. If you intend to build an application which will query the API extensively, please e-mail us at newsapps@tribune.com and we will hook you up with an API key. We place no limitations on the usage of the service beyond this, however, this implementation will allow us to take appropriate action in the event that a single source of the traffic is leading the service to be unstable.
